Authors: Seth Amico and John Teal

Date: 10 June 2026

## Project Overview

This project implements a small programmable processor in SystemVerilog for the TCES 330 final project. The design includes a control unit, finite state machine, program counter, instruction register, register file, ALU, RAM, ROM, seven-segment display decoder, and top-level DE-board wrapper.

The processor executes instructions stored in ROM and uses RAM for data memory. The main test program loads values from RAM, performs arithmetic through the ALU, stores the final result back to RAM, and then halts.

Final expected processor result:

    D[6A] = D[1B] - D[2A] + D[3C] - D[7E]
    D[6A] = 21BA - A04E + 71AC - B17F
    D[6A] = 4199

## Main Source Files

Project.sv  
Top-level board wrapper for the DE board. It connects the processor to switches, keys, LEDs, and HEX displays.

Processor.sv  
Top-level processor module. It connects the Control Unit and Datapath together.

Control_Unit.sv  
Contains the program counter, instruction register, ROM interface, and FSM control logic.

FSM.sv  
Finite state machine that decodes instructions and generates control signals for the datapath.

Datapath.sv  
Contains the register file, ALU, RAM wrapper, mux logic, and datapath testbench.

PC.sv  
Program counter module. Supports clear, increment, and jump-load behavior.

IR.sv  
Instruction register module. Loads a 16-bit instruction when IR_ld is active.

Decoder.sv  
Seven-segment HEX display decoder for active-low DE-board HEX displays.

myROM.v  
LPM ROM generated by Quartus. Stores the processor program.

myRAM.v  
LPM RAM generated by Quartus. Stores data memory values.

KeyFilter.sv  
Clock/key filtering module used for board-level stepping.

testProcessor.sv  
Provided or top-level processor simulation testbench used to demonstrate the full processor program.

## Memory Initialization Files

mifROM.mif  
Initializes ROM with the processor instruction program.

mifRAM.mif  
Initializes RAM with the data values used by the program.

Important RAM values used by the main program:

    RAM[1B] = 21BA
    RAM[2A] = A04E
    RAM[3C] = 71AC
    RAM[7E] = B17F

The processor stores the final computed result at:

    RAM[6A] = 4199

## Supported Instructions

Base project instructions:

    0000 = NOOP
    0001 = STORE
    0010 = LOAD
    0011 = ADD
    0100 = SUB
    0101 = HALT

Extra-credit jump instructions implemented in the FSM:

    1001 = JMP
    1010 = JNZ
    1011 = JLT

JMP performs an absolute jump.  
JNZ checks the ALU zero flag and jumps when the selected register is not zero.  
JLT checks signed less-than using the ALU negative and overflow flags.

The FSM testbench verifies the control behavior for JMP, JNZ taken, JNZ not taken, JLT taken, and JLT not taken. The main ROM program demonstrates the base arithmetic/load/store/halt processor sequence.

## How to Run the Full Test Suite

From the ModelSim project directory, run:

    do run_all_tests.do

Expected passing summaries include:

    PC_tb: 8 passes, 0 failures
    IR_tb: 4 passes, 0 failures
    Decoder_tb: 16 passes, 0 failures
    FSM_tb: 123 passes, 0 failures
    myRAM_tb: 5 passes, 0 failures
    Datapath_tb: 11 passes, 0 failures
    Control_Unit_tb: 5 passes, 0 failures
    Processor_tb: 4 passes, 0 failures

The main testProcessor simulation should complete normally and show the processor reaching the store operation with ALU result 4199.

## Expected Main Processor Trace

Important expected ALU results during the main program:

    21BA - A04E = 816C
    816C + 71AC = F318
    F318 - B17F = 4199

During the STORE instruction, the processor should show:

    D_Addr = 6A
    D_wr = 1
    ALU_A = 4199

The processor then reaches the HALT instruction:

    IR_Out = 5000
    State = S_HLT

## Notes About ModelSim Output

ModelSim may print messages such as:

    Break in Module ... at line ...

when a testbench reaches a finish statement. This is normal and does not indicate a design failure. The important transcript lines are the compile results, PASS/FAIL summaries, and whether the simulation completes as expected.

## Board Notes

The DE-board HEX displays are active-low. In the seven-segment decoder:

    0 = segment on
    1 = segment off

The processor uses a slowed/filtered clock for board-level stepping. Reset is active-low at the processor top level through ResetN and inverted internally for the active-high control reset.
